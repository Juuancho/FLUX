## Metrics

FLUX tracks four primary performance metrics. Each metric is defined precisely so that future measurements remain comparable.

**Throughput** — frames per second processed end-to-end, from `RawFrame` entering Queue A to `ProcessedFrame` exiting the ML Bridge as part of a complete batch. Measured on a steady-state pipeline (after warm-up) over a fixed number of frames.

**Latency per frame** — wall-clock time between a frame entering Phase A and its corresponding batch becoming available at the consumer. Reported as the median and the 95th percentile.

**Peak memory** — maximum resident memory of the FLUX process across the entire run, measured at one-second intervals. Reported separately for CPU memory and (when applicable) GPU memory.

**GPU idle fraction** — fraction of time the consuming GPU is not active, measured by sampling GPU utilization at 10 Hz during a training loop consuming the FLUX-produced DataLoader. Lower is better; values below 5% indicate the pipeline successfully keeps the GPU saturated.

These four metrics jointly characterize a pipeline configuration. Optimizing one in isolation often degrades another (for example, larger batch sizes improve GPU utilization but increase peak memory), so trade-offs are made explicit in the [Tuning Guide](#tuning-guide).

## Reference Hardware

Benchmarks are conducted on three reference hardware configurations representing the realistic deployment range of FLUX.

**Tier 1 — Consumer laptop.** NVIDIA GTX 1650 (4 GB VRAM), Intel Core i5 or equivalent (4 cores), 16 GB RAM, NVMe SSD. This is the development hardware and the lower bound of supported configurations.

**Tier 2 — Workstation.** NVIDIA RTX 3090 (24 GB VRAM) or equivalent, 8-core CPU, 64 GB RAM, NVMe SSD. Representative of researcher workstations.

**Tier 3 — Server-class GPU.** NVIDIA A100 (40 or 80 GB VRAM), 16+ core CPU, 128 GB RAM, fast NVMe storage. Representative of cluster nodes used for production training.

All benchmarks report the hardware tier explicitly. Numbers from different tiers are not comparable in absolute terms; only relative speedups (e.g. simple vs fast mode) are.

## Reference Datasets

Two datasets are used as benchmark inputs.

**`deeplense_classification_small`** — 1,000 frames sampled from the DeepLense Common Test I dataset. Single channel, 150×150 pixels, 3 classes (no, sphere, vort). Used for end-to-end classification throughput benchmarks.

**`mock_butler_dp0_subset`** — 1,000 frames generated by `MockButlerSource` from the DP0 manifest. Three channels (g, r, i), 64×64 pixels. Used for benchmarks involving the LSST-shaped data path.

Both datasets are small enough to fit comfortably in memory on Tier 1 hardware, allowing the benchmark to focus on pipeline overhead rather than I/O throughput. Larger-scale benchmarks (10,000 and 100,000 frames) are run on Tier 2 and Tier 3 hardware to validate that throughput scales linearly with dataset size.

## Methodology

Every reported number follows this protocol:

**Warm-up.** The first 50 frames of every run are excluded from measurements. This skips the cost of process initialization, JIT compilation, file system cache warm-up, and the pipeline's startup phase.

**Repetition.** Each measurement is repeated 5 times. The reported number is the median; the spread (max - min) is reported alongside as a sanity check on stability.

**Isolation.** Benchmarks are run on otherwise idle machines. Background processes that compete for CPU, memory, or I/O are terminated before a benchmark run.

**Determinism.** The pipeline is run with fixed random seeds (when applicable), and the test datasets are checksummed. Reproducing a benchmark requires only the same hardware tier, the same FLUX version, and the same dataset checksum.

**Reporting.** Each entry in the tables below cites the FLUX version, the hardware tier, and the dataset used. Results from different versions are not aggregated into single cells.

## Tuning Guide

The performance configuration fields and their trade-offs.

### `pipeline.queue_a_size` and `pipeline.queue_b_size`

Larger queues reduce the chance of stalls when one phase momentarily lags another. Smaller queues reduce memory and surface bottlenecks earlier.

Recommended values: `queue_a_size: 64`, `queue_b_size: 32` for Tier 1; double those for Tier 2 and Tier 3. Values above 256 rarely help and consume substantial memory when frames are large.

### `preprocessor.dask_scheduler`

`threads` is the right default for FLUX's filters because all five built-in filters release the GIL during their NumPy operations. `processes` is slower than `threads` for typical workloads due to the cost of serializing image data across process boundaries; it is only beneficial when a custom filter holds the GIL extensively.

### `preprocessor.num_workers`

`0` (auto-detect as `cpu_count() - 1`) is the right default. Manually setting a higher value than `cpu_count()` causes thread thrashing without gaining throughput.

### `preprocessor.chunk_size`

The number of frames per Dask task. Smaller chunks balance load better across workers but increase scheduling overhead. Larger chunks reduce overhead but can leave workers idle if they finish at different times. Default of 16 is a good starting point. Tune by measuring throughput at 8, 16, 32, 64.

### `ml_bridge.prefetch_factor`

Prefetching is the primary lever for keeping the GPU busy. Increase `prefetch_factor` until GPU idle fraction drops below 5% or memory becomes the binding constraint.

Memory cost of prefetching: `prefetch_factor × batch_size × frame_size_in_bytes`. For a ResNet18 batch of 32 frames at 64×64×3 float32, this is `prefetch_factor × 1.5 MB` — small in absolute terms.

### `ml_bridge.batch_size`

Larger batches improve throughput up to the point where the GPU becomes the bottleneck. Beyond that, larger batches just increase memory without speedup. Tune by sweeping `batch_size` from 16 up to the largest value that fits in GPU memory.

For training, larger batch sizes also affect optimization dynamics — that is a model concern, not a FLUX concern, but worth keeping in mind.

The expected results, based on the architectural analysis:

- FLUX `simple` should match RIPPLe within ±10% (same essential algorithm)
- FLUX `fast` should achieve 4–8x higher throughput than RIPPLe
- FLUX should consume slightly more memory than RIPPLe (queues, prefetch buffer) but the increase should be bounded and predictable
- FLUX `fast` should reduce GPU idle from RIPPLe's typical 30–40% to under 5%

If measured results contradict any of these expectations significantly, that is grounds for revisiting the design decisions in [DESIGN_PHILOSOPHY.md](DESIGN_PHILOSOPHY.md). The architecture exists to deliver these gains; if it does not, it should be reconsidered.
